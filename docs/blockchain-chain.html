On-Chain Blockchain

A Blockchain Within a Blockchain

Blockchella implements its own mini-blockchain inside the Ethereum smart contract. Every time a group successfully completes and tickets are distributed, a new block is appended to this chain — creating a permanent, tamper-evident, cryptographically linked record of every ticket group that ever completed.

This is a blockchain running on top of Ethereum — a chain within a chain.

Why Build This?

Ethereum already stores all transaction history, so why build a separate chain? A few reasons:

  • Custom data structure — each block stores exactly the information relevant to ticketing: group ID, member addresses, ticket count, and timestamp.
  • Cryptographic integrity — each block commits to the hash of the previous block, so you can verify the entire history hasn't been tampered with in a single call.
  • Educational demonstration — Blockchella is a proof-of-concept that demonstrates core blockchain concepts (hashing, linking, genesis block, integrity verification) implemented in Solidity.
  • Queryable history — the admin panel can read and display every block directly from the contract without needing event indexing or a backend.

The Genesis Block

Like Bitcoin and Ethereum themselves, Blockchella's chain begins with a genesis block — block number 0 — created at contract deployment. It is a placeholder with no group data: 

Block #0 (Genesis)
  groupId:      0
  previousHash: 0x0000...0000
  blockHash:    keccak256("BlockFest Genesis")
  members:      []
  ticketCount:  0
  timestamp:    deployment time

Every subsequent block's previousHash links back to this origin, forming an unbroken chain.

How Blocks Are Linked

Each block's hash is computed by hashing its own data together with the previous block's hash. This means changing any block in history would change its hash, which would break the link to the next block, and so on — making tampering immediately detectable. 

┌──────────────────────────┐
│  Block #0  (Genesis)     │
│  hash: 0xABCD...         │
└────────────┬─────────────┘
             │ previousHash
┌────────────▼─────────────┐
│  Block #1                │
│  groupId: 3              │
│  members: [0x111, 0x222] │
│  hash: keccak256(        │
│    1, 3, 0xABCD...,      │
│    timestamp,            │
│    members, 2            │
│  ) = 0xEF01...           │
└────────────┬─────────────┘
             │ previousHash
┌────────────▼─────────────┐
│  Block #2                │
│  groupId: 7              │
│  members: [0x333]        │
│  hash: keccak256(        │
│    2, 7, 0xEF01...,      │
│    timestamp,            │
│    members, 1            │
│  ) = 0x2345...           │
└──────────────────────────┘

What Each Block Contains

blockNumber Zero-based position in the chain. Block 0 is genesis.
groupId The Blockchella group ID that this block represents.
previousHash The blockHash of the block immediately before this one in the chain.
blockHash A keccak256 hash committing to this block's position, group, previous hash, timestamp, members, and ticket count.
members Array of wallet addresses that received tickets in this group.
timestamp The Ethereum block timestamp when this chain block was added.
ticketCount Number of tickets distributed in this group (equals the number of members).

Verifying Chain Integrity

The verifyChain() function iterates every block from index 1 to the current chain length and checks that each block's previousHash matches the actual blockHash of the block before it.

If all links match, it returns true — the chain is intact and has not been tampered with. If any link is broken, it returns false.

You can trigger this check from the Admin Panel using the Verify Chain button, which calls the function as a read-only view (no gas cost, no transaction needed).

⚠️ Gas warning: verifyChain() contains an unbounded loop. It is safe to call as a view function off-chain (no gas cost), but must never be called inside a transaction or from another contract — it would exceed the block gas limit as the chain grows.

Why This Is Meaningful

Ethereum is itself a blockchain. Every transaction that calls Blockchella's contract is already recorded permanently on the Ethereum chain. So Blockchella's internal chain is a deliberate demonstration — it shows that the same fundamental concepts (cryptographic hashing, linked blocks, a genesis block, tamper detection) can be implemented in application-layer code running on top of another blockchain.

It is a blockchain that understands it is a blockchain.
