Smart Contract

BlockFest Contract

The Blockchella smart contract is the heart of the system. It is written in Solidity, deployed on Ethereum Sepolia, and handles everything: group creation, escrow, ticket distribution, refunds, and the on-chain blockchain record. No backend server is involved — all logic runs on-chain.

What is a Smart Contract?

A smart contract is a program that lives on the blockchain. Once deployed, it runs exactly as written — no one can alter it, no company controls it, and it executes automatically when its conditions are met. Blockchella uses a smart contract so that the ticketing rules are enforced by code, not by trust in a company.

ERC-20 Ticket Tokens

Blockchella implements the ERC-20 token standard, but with one key restriction: tickets can only be transferred back to the contract owner (the venue). This prevents scalping and secondary market resales. Each wallet can hold at most one ticket, represented as a balance of 1 in the ERC-20 token mapping.

The token is named BlockFest Ticket with the symbol BFT and zero decimal places — tickets are whole units only, you cannot own half a ticket.

Key Functions

createGroup(address[] members, uint256 deadlineHours)

Creates a new group with the caller as creator plus any additional member addresses. Pass an empty array to create a solo group (just yourself). The deadline is set in hours from the current block timestamp — between 1 and 168 hours. Everyone named in the group must pay before the deadline or all funds are refunded.

joinGroup(uint256 groupId)

Called by a group member to pay their ticket price into escrow. Must send exactly the ticket price in SETH as the transaction value. On the first payment, the contract reserves tickets for the entire group. When the last member pays, tickets are automatically distributed to all members and the group is marked complete.

returnTicket()

Allows a ticket holder to return their ticket for a full refund of the ticket price. The ticket token is burned and SETH is sent back to the caller's wallet. Gas fees are not refunded.

refundGroup(uint256 groupId)

Can be called by anyone once a group's deadline has passed without all members paying. Sets the group to Cancelled and refunds all members who had paid into escrow. This lets anyone trigger the cleanup, not just the group creator.

cancelGroup(uint256 groupId)

Only the group creator or contract owner can call this before the deadline. Cancels an active group and refunds all members who have paid. Useful if the creator realises the group won't fill up.

claimRefund()

If an automatic refund push failed (rare edge case, e.g. the recipient is a contract that rejects ETH), the unclaimed amount is recorded and the user can call this to pull their refund manually.

availableTickets()

A read-only view function that returns how many tickets are still available. Computed as totalTickets − reservedTickets − soldTickets. The frontend calls this without needing MetaMask connected.

verifyChain()

Iterates every block in the on-chain chain and checks that each block's previousHash matches the actual hash of the block before it. Returns true if all links are valid. Intended for off-chain use only — calling it in a transaction would use too much gas as the chain grows.

Security Features

ReentrancyGuard
Inherited from OpenZeppelin. The nonReentrant modifier on all value-transferring functions prevents a malicious contract from calling back into Blockchella mid-execution to drain funds.
Checks-Effects-Interactions
All functions validate inputs first (Checks), then update contract state (Effects), then make external calls like ETH transfers (Interactions). This ordering prevents state from being manipulated by reentrant calls.
Pull Refund Fallback
If a push refund fails (e.g. the recipient contract rejects ETH), the amount is saved and the user can pull it with claimRefund(). No funds are ever permanently lost.
Access Control
Owner-only functions (like withdrawFunds) are protected by the onlyOwner modifier. Group cancel is restricted to the group creator or contract owner via onlyCreatorOrOwner.
Escrow Accounting
withdrawFunds can only withdraw funds not locked in active escrow or unclaimed refunds. The owner can never accidentally take money belonging to active groups.

Deployment Details

The contract is deployed with two constructor arguments: the ticket price in wei and the total ticket supply. Both are fixed at deployment and can never be changed. The entire ticket supply is minted to the owner's wallet at deployment; tickets are transferred from the owner's pool to members when a group completes — the total supply never increases.
